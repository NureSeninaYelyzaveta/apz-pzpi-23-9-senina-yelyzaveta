Розділ 1. Вступ
	У сучасних умовах стрімкого розвитку інформаційних технологій особливого значення набуває створення ефективних web-застосунків, що забезпечують зручну взаємодію користувачів із програмними системами у режимі реального часу. Використання web-інтерфейсів дозволяє реалізувати доступ до функціональних можливостей системи незалежно від платформи та місця перебування користувача, що є важливим фактором у сфері освітніх інформаційних систем.
	Актуальність розробки клієнтської частини програмних систем зумовлена необхідністю забезпечення інтуїтивно зрозумілого, функціонального та адаптивного інтерфейсу користувача, який дозволяє ефективно взаємодіяти з серверною частиною системи. Особливу увагу при цьому приділяють підтримці різних ролей користувачів, організації доступу до даних відповідно до прав користувача, а також забезпеченню можливостей адміністрування системи.
	У межах даної лабораторної роботи реалізується front-end частина web-застосунку для інформаційної системи ArtProgress, яка призначена для організації взаємодії між учасниками освітнього процесу. Розроблений web-застосунок забезпечує роботу користувачів у веб-середовищі та підтримує декілька ролей, зокрема: студент, викладач, батьки та адміністратор. Для кожної ролі передбачено окремий інтерфейс, що відповідає функціональним потребам відповідного користувача.
	Клієнтська частина системи взаємодіє із серверною частиною, реалізованою засобами платформи C#, за допомогою HTTP-запитів до API. Обмін даними здійснюється у форматі JSON, що забезпечує стандартизовану передачу інформації між компонентами системи. Для реалізації клієнтської частини використано мови розмітки та програмування, зокрема HTML для створення структури сторінок, CSS для їх стилізації та JavaScript для реалізації динамічної поведінки інтерфейсу та обробки даних.
	Важливою складовою розробленого web-застосунку є підтримка механізмів авторизації користувачів, збереження сесійних даних у локальному сховищі браузера, а також реалізація переходів між сторінками відповідно до ролі користувача. Крім того, передбачено базові елементи інтернаціоналізації, зокрема використання локалізованих форматів дати та часу.
	Адміністративна частина системи забезпечує можливість управління даними, зокрема додавання та видалення дисциплін, а також отримання узагальненої статистики щодо стану системи. У разі відсутності доступу до серверної частини передбачено використання локального сховища як альтернативного механізму збереження даних, що підвищує надійність роботи застосунку.
	Метою даної лабораторної роботи є розробка клієнтської частини web-застосунку, що забезпечує підтримку багаторольового доступу, взаємодію з серверною частиною системи, а також реалізацію базових функцій адміністрування та обробки даних.
	Для досягнення поставленої мети необхідно вирішити такі завдання: реалізувати інтерфейс авторизації користувачів; забезпечити розмежування доступу відповідно до ролей; розробити інтерфейси для студентів, викладачів, батьків та адміністратора; організувати взаємодію з API серверної частини; реалізувати обробку та відображення даних у web-інтерфейсі; забезпечити базові механізми локалізації та збереження даних на стороні клієнта.


Розділ 2. Опис предметної області
	У межах даної лабораторної роботи розглядається предметна область, пов’язана з організацією та підтримкою освітнього процесу за допомогою інформаційної системи ArtProgress. Основним призначенням такої системи є забезпечення ефективної взаємодії між учасниками освітнього процесу, а саме студентами, викладачами, батьками та адміністраторами, а також автоматизація процесів обліку навчальних досягнень, контролю успішності та управління навчальними даними.
	Інформаційна система ArtProgress орієнтована на централізоване зберігання та обробку даних про користувачів, навчальні дисципліни та результати навчання. У системі передбачено ведення обліку студентів, викладачів та батьків, а також зберігання інформації про оцінки, що виставляються в межах навчального процесу. Це дозволяє забезпечити прозорість оцінювання, своєчасний доступ до інформації та підвищити ефективність контролю знань.
	Особливістю предметної області є наявність декількох категорій користувачів, кожна з яких має власні функціональні можливості та рівень доступу до даних. Студент у межах системи має можливість переглядати власні оцінки, отримувати інформацію про навчальні дисципліни, а також взаємодіяти з елементами власного профілю. Додатково передбачено можливість формування персонального портфоліо, що дозволяє зберігати результати творчої або навчальної діяльності у вигляді зображень та текстових описів.
	Викладач, у свою чергу, виконує функції оцінювання та контролю успішності студентів. У системі для нього реалізовано можливість пошуку студентів, вибору навчальних дисциплін та додавання оцінок із фіксацією відповідних параметрів. Крім того, викладач має доступ до перегляду останніх виставлених оцінок, що дозволяє здійснювати оперативний контроль за навчальним процесом та актуальністю даних.
	Батьки виступають як користувачі, що мають обмежений доступ до інформації, пов’язаної з навчальними досягненнями студентів. Основною функцією для цієї категорії є перегляд оцінок дитини, аналіз її успішності та отримання узагальнених показників, зокрема середнього балу. Такий підхід забезпечує можливість контролю навчального процесу з боку батьків без втручання у внутрішні механізми системи.
	Адміністратор є ключовим користувачем, який відповідає за управління системою в цілому. До його функціональних обов’язків належить управління даними, зокрема створення та видалення навчальних дисциплін, а також отримання статистичної інформації щодо стану системи. Адміністративний інтерфейс дозволяє аналізувати кількість користувачів різних категорій, кількість оцінок, дисциплін та інші узагальнені показники, що є важливими для контролю функціонування системи.
	Важливим аспектом предметної області є взаємозв’язок між її основними сутностями. Студенти пов’язані з батьками через відповідні відношення, що дозволяє реалізувати доступ до даних про успішність. Викладачі взаємодіють зі студентами шляхом виставлення оцінок у межах певних дисциплін. Дисципліни виступають як окремі об’єкти системи, що визначають контекст оцінювання. Оцінки, у свою чергу, є результатом взаємодії між студентом, викладачем та навчальною дисципліною і містять інформацію про значення оцінки та дату її встановлення.
	Предметна область також передбачає використання механізмів обробки даних, зокрема отримання, збереження та відображення інформації у web-інтерфейсі. Дані надходять із серверної частини системи у вигляді структурованих повідомлень та обробляються на стороні клієнта для подальшого відображення користувачеві. Важливою складовою є також забезпечення актуальності даних та обробка можливих помилок під час взаємодії з сервером.
	Таким чином, предметна область даної роботи охоплює комплекс процесів, пов’язаних із управлінням освітніми даними, організацією доступу до них відповідно до ролей користувачів та забезпеченням ефективної взаємодії між усіма учасниками освітнього процесу за допомогою web-застосунку.
Розділ 3. Опис взаємодії компонентів web-застосунку
	3.1 Загальна архітектура взаємодії компонентів
	Розроблений web-застосунок реалізовано за клієнт-серверною архітектурою, у якій front-end частина взаємодіє з серверною частиною через HTTP-запити до REST API. Основною метою такої архітектури є розділення відповідальності між інтерфейсом користувача та обробкою даних.
	Клієнтська частина застосунку реалізована з використанням мов розмітки та програмування HTML, CSS та JavaScript і виконується у веб-браузері користувача. Вона забезпечує відображення інтерфейсу, обробку дій користувача, формування запитів до серверу та обробку отриманих відповідей.
	Серверна частина, реалізована раніше із використанням платформи C#, виконує функції обробки запитів, взаємодії з базою даних та повернення результатів у форматі JSON. У межах даної лабораторної роботи серверна частина розглядається лише як зовнішній компонент, що надає API для клієнтського застосунку.
	Взаємодія між компонентами здійснюється за наступною схемою:
	- користувач взаємодіє з інтерфейсом web-сторінки;
	- JavaScript-код обробляє події (натискання кнопок, введення даних тощо);
	- формується HTTP-запит до відповідного API-методу;
	- сервер обробляє запит та повертає відповідь у форматі JSON;
	- клієнтська частина отримує дані та динамічно оновлює DOM-структуру сторінки.
	Залежно від ролі користувача (студент, викладач, батьки, адміністратор) система надає доступ до різних інтерфейсів, що реалізовано шляхом перенаправлення на відповідні HTML-сторінки після проходження авторизації. Роль користувача визначається під час входу до системи та зберігається у локальному сховищі браузера (localStorage), що забезпечує доступ до відповідних даних на інших сторінках застосунку.
	Архітектура застосунку передбачає використання окремих модулів JavaScript для кожної ролі користувача. Кожен модуль відповідає за обробку подій, взаємодію з API та відображення даних у межах конкретної сторінки. Такий підхід забезпечує логічне розділення функціональності та спрощує підтримку програмного коду.
	Для обміну даними використовується формат JSON, що є стандартним для web-застосунків та забезпечує зручність серіалізації і десеріалізації даних між клієнтом і сервером.
	Таким чином, взаємодія компонентів web-застосунку організована за принципами клієнт-серверної архітектури з використанням REST API, що забезпечує гнучкість, масштабованість та зручність подальшого розвитку системи.

	3.2 Організація авторизації та маршрутизації користувачів
	У розробленому web-застосунку механізм авторизації користувачів реалізовано на клієнтському рівні з використанням JavaScript та взаємодії з серверним API. Основною метою авторизації є ідентифікація користувача та визначення його ролі в системі для подальшого надання відповідного функціоналу.
	Сторінка авторизації є початковою точкою входу до системи та містить форму введення даних користувача, зокрема електронної пошти та вибору ролі. Обробка події надсилання форми здійснюється за допомогою JavaScript, що дозволяє перехопити стандартну поведінку браузера та виконати перевірку введених даних.
	Після введення користувачем необхідних даних формується запит до відповідного API-методу залежно від обраної ролі. Для кожного типу користувача використовується окремий endpoint, що дозволяє отримати дані конкретного користувача за електронною поштою. У разі успішного отримання відповіді виконується пошук відповідного об’єкта користувача та перевірка коректності введених даних. У випадку, якщо користувач з вказаними даними не знайдений або сервер повертає помилку, система відображає повідомлення про помилку. Це забезпечує базову валідацію та запобігає несанкціонованому доступу до функціоналу системи.
	Після успішної авторизації дані користувача, такі як ідентифікатор, ім’я, електронна пошта та роль, зберігаються у локальному сховищі браузера (localStorage). Це дозволяє використовувати ці дані на інших сторінках застосунку без необхідності повторного звернення до серверу при кожному переході.
	Маршрутизація користувачів реалізована у вигляді перенаправлення на відповідні HTML-сторінки залежно від ролі користувача. Перенаправлення виконується шляхом зміни значення властивості window.location.href, що забезпечує завантаження відповідної сторінки інтерфейсу.
	Особливістю реалізації є окрема логіка авторизації адміністратора. Доступ до адміністративної панелі здійснюється на основі попередньо визначених облікових даних, що перевіряються безпосередньо на клієнтській стороні. У разі успішної перевірки виконується очищення попередніх даних у localStorage та запис нових значень, після чого відбувається перенаправлення до адміністративного інтерфейсу.
	Додатково реалізовано можливість реєстрації нових користувачів. Для цього передбачено окрему форму, яка динамічно відображається замість форми авторизації. Після заповнення даних формується POST-запит до відповідного API-методу створення користувача залежно від обраної ролі. У разі успішного виконання операції система повідомляє користувача про успішну реєстрацію та повертає його до форми авторизації.
	Таким чином, механізми авторизації та маршрутизації забезпечують коректний розподіл доступу до функціональних можливостей системи відповідно до ролей користувачів та реалізують зручну навігацію між сторінками web-застосунку.

	3.3 Взаємодія з API та обробка даних
	У розробленому web-застосунку взаємодія клієнтської частини із серверною реалізована на основі архітектурного підходу REST API. Обмін даними здійснюється через HTTP-запити, які формуються та обробляються на стороні клієнта за допомогою JavaScript. Основним інструментом взаємодії виступає функція fetch, що забезпечує асинхронне отримання та відправлення даних без необхідності перезавантаження сторінки.
	Передача даних між клієнтом і сервером виконується у форматі JSON, що є стандартом для сучасних web-застосунків. Це дозволяє легко серіалізувати та десеріалізувати структури даних, а також забезпечує сумісність між різними частинами системи.
	У межах застосунку реалізовано повний цикл CRUD-операцій:
	- отримання даних (читання) через GET-запити;
	- створення нових сутностей через POST-запити;
	- видалення даних через DELETE-запити;
	- часткове оновлення даних (в окремих сценаріях через повторну відправку оновлених об’єктів).
	GET-запити використовуються для завантаження основних даних системи, зокрема списків студентів, викладачів, дисциплін, оцінок та профільної інформації користувачів. Отримані дані далі використовуються для формування інтерфейсу відповідної сторінки.
	POST-запити застосовуються у сценаріях створення нових записів у системі. Наприклад, під час реєстрації користувача формується об’єкт із введених даних та надсилається на сервер, де створюється новий запис у базі даних. Аналогічно реалізовано додавання оцінок викладачем та створення нових дисциплін адміністратором.
	DELETE-запити використовуються в адміністративній частині системи для видалення записів, зокрема дисциплін. Після виконання операції відбувається повторне завантаження даних для актуалізації інтерфейсу.
	Важливою особливістю реалізації є асинхронна обробка запитів. Використання конструкцій async/await дозволяє організувати послідовне виконання операцій та зробити код більш читабельним. Для обробки можливих помилок застосовується блок try/catch, що дозволяє перехоплювати помилки мережевого з’єднання або некоректні відповіді сервера.
	Після отримання даних із API виконується їх обробка на стороні клієнта. Обробка включає трансформацію даних у зручний для відображення формат та підготовку до рендерингу в DOM. Зокрема, здійснюється:
	- фільтрація даних відповідно до ролі користувача;
	- сортування та обмеження вибірки (наприклад, відображення останніх оцінок);
	- обчислення агрегованих значень, таких як середній бал;
	- форматування дат у локальний вигляд через toLocaleDateString() та toLocaleString().
	Відображення даних реалізовано через динамічне формування HTML-структури. JavaScript створює DOM-елементи або змінює вміст існуючих елементів сторінки, після чого дані відображаються користувачу без перезавантаження сторінки. Такий підхід забезпечує інтерактивність інтерфейсу та підвищує швидкість реакції системи.
	Окремо у застосунку використовується локальне сховище браузера (localStorage), яке виконує допоміжну роль у збереженні даних на стороні клієнта. У ньому зберігаються:
	- ідентифікатор користувача;
	- роль користувача;
	- базова профільна інформація;
	- тимчасові дані інтерфейсу (наприклад, зображення профілю або портфоліо студента).
	У випадках, коли серверна частина недоступна або запит не може бути виконаний, частково використовується резервна логіка з localStorage, що дозволяє зберегти базову працездатність окремих функцій інтерфейсу.
	Таким чином, взаємодія з API та обробка даних у web-застосунку побудовані на принципах асинхронної комунікації, клієнтської обробки JSON-даних та динамічного оновлення інтерфейсу, що забезпечує гнучкість, швидкодію та зручність використання системи.

	3.4 Опис інтерфейсів користувачів за ролями
	У розробленому web-застосунку реалізовано багаторівневу рольову модель доступу, яка визначає функціональні можливості користувача залежно від його типу в системі. Такий підхід забезпечує розмежування доступу до даних та операцій, а також підвищує безпеку та структурованість взаємодії з системою.
	Кожен тип користувача має окремий інтерфейс, який реалізовано у вигляді окремої HTML-сторінки з власною логікою JavaScript та стилізацією. Перехід між інтерфейсами здійснюється після проходження авторизації та визначення ролі користувача.
		3.4.1 Інтерфейс авторизації та реєстрації
		Сторінка авторизації є початковою точкою входу до системи та виконує функцію ідентифікації користувача. Інтерфейс містить форму введення основних даних (ім’я, електронна пошта) та вибір ролі, що визначає подальшу поведінку системи.
	На основі введених даних виконується запит до відповідного API, після чого система перевіряє наявність користувача в базі даних. У разі успішної авторизації виконується збереження даних у локальному сховищі браузера та перенаправлення на відповідну сторінку.
	Окремо реалізовано механізм реєстрації, який дозволяє створювати нові облікові записи. Форма реєстрації є динамічною та відображається замість форми входу. Після заповнення даних формується запит на сервер для створення нового користувача відповідного типу.
	Після успішної реєстрації система повертає користувача до форми авторизації, що забезпечує єдиний сценарій входу до системи.

У таблиці відображаються дисципліни, отримані оцінки, викладачі та дати їх виставлення, що дозволяє отримати повну картину навчальної активності. Це забезпечує масштабованість інтерфейсу та зручність використання системи в багатодітних профілях.                                                                                         
		Таким чином, інтерфейс батьків реалізує гнучку структуру відображення даних, яка адаптується до кількості пов’язаних студентів і забезпечує зручний доступ до навчальної інформації кожної дитини окремо.

Крім того, застосування динамічного оновлення інтерфейсу сприяє підвищенню інтерактивності та продуктивності роботи системи.

	3.5 Реалізація взаємодії з серверною частиною
	Взаємодія клієнтської частини вебзастосунку із серверною реалізована на основі архітектурного підходу REST (Representational State Transfer). Серверна частина надає набір API-ендпоінтів, до яких здійснюються HTTP-запити з фронтенду. Передача даних виконується у форматі JSON, що забезпечує уніфікований спосіб обміну інформацією між компонентами системи.
	Основним інструментом для взаємодії з сервером у клієнтській частині є вбудований механізм fetch, який дозволяє виконувати асинхронні запити без необхідності перезавантаження сторінки. Це забезпечує динамічність інтерфейсу та покращує користувацький досвід.
	Для централізації доступу до серверних ресурсів використовується базова змінна, яка містить адресу API:
const API_URL = "https://localhost:7033/api";
	Отримання даних з сервера
	Отримання інформації реалізовано через GET-запити до відповідних API-ендпоінтів. 	Наприклад, у профілі батьків здійснюється завантаження даних про батька за його ідентифікатором:
const res = await fetch(`${API_URL}/parents/${parentId}`);
const parent = await res.json();
	Отримані дані використовуються для заповнення інтерфейсу користувача, зокрема відображення імені та контактної інформації.
	Подібний підхід використовується і для отримання інформації про студента та його оцінки:
const studentRes = await fetch(`${API_URL}/students/${studentId}`);
const student = await studentRes.json();
	Передача даних на сервер
	Додавання нових сутностей у систему реалізовано через POST-запити. Наприклад, у модулі адміністратора реалізовано додавання нової дисципліни:
const res = await fetch(`${API_URL}/admin/disciplines`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(name)
});
Аналогічно у модулі викладача реалізовано додавання оцінки студенту:
const res = await fetch(`https://localhost:7033/api/grades`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(newGrade)
});
Такий підхід забезпечує можливість динамічного оновлення даних у системі без необхідності перезавантаження сторінки.
	Обробка відповідей сервера
	Після виконання запитів здійснюється перевірка статусу відповіді сервера. У разі успішного виконання запиту дані використовуються для оновлення інтерфейсу користувача, інакше виводиться повідомлення про помилку або застосовується резервна логіка.
	Приклад перевірки:
if (res.ok) {
    await loadAllData();
}
	У випадку недоступності серверної частини частково використовується локальне сховище браузера (localStorage) для збереження даних, що дозволяє забезпечити базову функціональність застосунку навіть без активного підключення до API.
	Взаємодія з різними модулями системи
	Кожна роль користувача (студент, викладач, батьки, адміністратор) взаємодіє із серверною частиною через окремі API-запити. Наприклад:
	- студент отримує свої оцінки через /students/{id}
	- викладач працює з /grades та /students
	- батьки отримують дані через зв’язок ParentStudents
	- адміністратор керує дисциплінами через /admin/disciplines
	Такий підхід забезпечує чітке розмежування доступу до даних відповідно до ролей користувачів.
	Реалізований механізм взаємодії з серверною частиною базується на використанні REST API та асинхронних HTTP-запитів. Це забезпечує гнучкість, масштабованість системи та можливість динамічного оновлення даних інтерфейсу без перезавантаження сторінок. Використання JSON як основного формату обміну даними дозволяє забезпечити сумісність між клієнтською та серверною частинами вебзастосунку.

	3.6 Реалізація збереження даних на стороні клієнта
	У вебзастосунку реалізовано механізм збереження даних на стороні клієнта з використанням Web Storage API, зокрема технології localStorage. Даний механізм застосовується для зберігання даних, які не потребують постійної синхронізації із серверною частиною, але є критично важливими для підтримки стану інтерфейсу та забезпечення безперервності користувацького сеансу.
	Основною функцією localStorage у системі є збереження даних авторизації користувача після успішного входу в систему. Зокрема, після перевірки користувача в залежності від його ролі в localStorage записуються такі параметри, як ідентифікатор користувача, ім’я, email та роль. Це дозволяє зберігати стан авторизації навіть після оновлення сторінки або повторного відкриття застосунку, без необхідності повторного входу.
localStorage.setItem("studentId", id);
localStorage.setItem("studentName", user.name);
localStorage.setItem("role", role);
	Використання ролі користувача є ключовим елементом маршрутизації інтерфейсу. На основі цього значення система визначає, який саме інтерфейс необхідно відобразити (студентський, викладацький, батьківський або адміністративний), що забезпечує розділення доступу відповідно до ролей.
	Окрім даних авторизації, у локальному сховищі також зберігаються допоміжні користувацькі дані, які покращують зручність роботи із системою. Зокрема, реалізовано збереження зображень профілю користувача. Для цього використовується формат Base64, що дозволяє зберігати зображення у вигляді текстового рядка без необхідності серверного сховища.
localStorage.setItem(`studentPhoto_${studentId}`, reader.result);
	Також у localStorage зберігаються додаткові персональні налаштування користувача, наприклад опис профілю або дані портфоліо студента. Це дозволяє зберігати індивідуальні зміни інтерфейсу без звернення до серверної частини.
	Таким чином, використання локального сховища забезпечує зменшення навантаження на сервер, підвищення швидкодії інтерфейсу та часткову автономність роботи вебзастосунку.

	3.7 Реалізація обробки та відображення даних
	Обробка даних у клієнтській частині вебзастосунку здійснюється після отримання інформації від серверного API у форматі JSON. Отримані дані підлягають подальшій трансформації у структури, придатні для відображення в інтерфейсі користувача.
	Основним механізмом відображення інформації є динамічне формування DOM-елементів на основі отриманих масивів даних. Зокрема, для відображення оцінок студентів використовується таблична структура, де кожен запис генерується програмно на основі даних, отриманих із сервера.
grades.forEach(g => {
    const row = `
        <tr>
            <td>${g.disciplineName}</td>
            <td>${g.value}</td>
            <td>${g.teacherName}</td>
            <td>${g.date}</td>
        </tr>
    `;
    table.innerHTML += row;
});
	Такий підхід дозволяє забезпечити динамічне оновлення інтерфейсу без необхідності перезавантаження сторінки. Оновлення даних реалізується через асинхронні запити, після яких виконується повторний рендеринг відповідних DOM-елементів.
	Додатково у системі реалізовано клієнтські обчислення, зокрема розрахунок середнього балу на основі отриманих оцінок. Обчислення виконується безпосередньо у браузері після завантаження даних:
const avg = (sum / grades.length).toFixed(2);
	Отриманий результат одразу відображається в інтерфейсі, що забезпечує інтерактивність та оперативність оновлення інформації. Таким чином, реалізований механізм обробки даних забезпечує ефективне перетворення серверної інформації у зручний для користувача вигляд.

	3.8 Реалізація інтерфейсу адміністратора
	Інтерфейс адміністратора є окремим функціональним модулем вебзастосунку, який призначений для централізованого управління даними системи. Доступ до адміністративної панелі реалізовано на основі ролі користувача, яка визначається під час процесу авторизації.
	Головною особливістю адміністративного інтерфейсу є наявність блоку статистики, який формує узагальнені показники стану системи. До таких показників належать кількість студентів, викладачів, дисциплін та оцінок, а також середній бал. Формування цих даних здійснюється на основі запитів до відповідних API-ендпоінтів та подальшої агрегації отриманої інформації на стороні клієнта.
document.getElementById("studentsCount").textContent = students.length;
document.getElementById("avgGrade").textContent = avg;
	Окремим функціональним компонентом є модуль управління дисциплінами. Він дозволяє адміністраторам додавати нові навчальні дисципліни та видаляти існуючі. Усі операції реалізовано через REST-запити до серверної частини, що забезпечує актуальність даних.
await fetch(`${API_URL}/admin/disciplines`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(name)
});
	У випадку недоступності серверної частини передбачено резервний механізм збереження даних у localStorage, що дозволяє зберігати частину функціональності адміністративного інтерфейсу. Оновлення інтерфейсу адміністратора відбувається динамічно після кожної операції (додавання або видалення даних), що виключає необхідність ручного перезавантаження сторінки та забезпечує актуальність відображуваної інформації.

Розділ 4. Опис взаємодії компонентів вебзастосунку
	Вебзастосунок реалізовано як клієнт-серверну систему, у якій логіка поділена між трьома основними рівнями: структурним (HTML), поведінковим (JavaScript) та серверним (REST API). Така архітектура забезпечує чітке розмежування відповідальності компонентів та дозволяє незалежно розвивати фронтенд і бекенд частини системи.

	4.1 Взаємодія HTML та JavaScript
	HTML у системі виконує роль статичної основи інтерфейсу користувача. Він визначає структуру сторінок для різних ролей користувачів, включаючи форми авторизації, профільні блоки, таблиці оцінок та адміністративні панелі. Кожна сторінка містить елементи з унікальними ідентифікаторами, які використовуються JavaScript для подальшого динамічного керування вмістом.
	JavaScript у даному застосунку реалізує логіку взаємодії користувача з інтерфейсом та відповідає за зміну стану сторінки без її перезавантаження. Основним механізмом взаємодії є DOM-маніпуляції, за допомогою яких відбувається динамічне оновлення текстових полів, таблиць та інших елементів інтерфейсу.
	Наприклад, після отримання даних про користувача система автоматично підставляє їх у відповідні HTML-елементи:
document.getElementById("userName").textContent = data.name;
document.getElementById("userRole").textContent = data.group || "-";
	Таким чином, HTML формує структуру інтерфейсу, тоді як JavaScript забезпечує його динамічність та інтерактивність.

	4.2 Взаємодія JavaScript із серверним API
	Основою обміну даними між клієнтською та серверною частинами є REST API. Усі операції отримання, додавання та оновлення даних реалізуються через HTTP-запити, які виконуються за допомогою функції fetch.
	Сервер надає набір ендпоінтів для роботи з основними сутностями системи: студентами, викладачами, батьками, оцінками та дисциплінами. Кожен запит виконується асинхронно, що дозволяє уникнути блокування інтерфейсу.
	Приклад отримання даних про студента:
const res = await fetch(`${API_URL}/students/${studentId}`);
const data = await res.json();
	
	Приклад відправлення даних на сервер при додаванні оцінки:
await fetch(`${API_URL}/grades`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(newGrade)
});
	Таким чином, JavaScript виступає проміжним шаром між користувацьким інтерфейсом та серверною логікою, забезпечуючи синхронізацію даних у реальному часі.

	4.3 Обробка та динамічне відображення даних
	Після отримання даних від серверної частини вони надходять у форматі JSON та обробляються на стороні клієнта. Подальша обробка полягає у перетворенні структурованих даних у візуальні елементи інтерфейсу.
	Основним механізмом відображення є динамічне формування DOM-елементів на основі отриманих масивів даних. Наприклад, таблиця оцінок студентів генерується автоматично:
grades.forEach(g => {
    const row = `
        <tr>
            <td>${g.disciplineName}</td>
            <td>${g.value}</td>
            <td>${g.teacherName}</td>
            <td>${g.date ? new Date(g.date).toLocaleDateString() : "-"}</td>
        </tr>
    `;
    table.innerHTML += row;
});

	Оновлення інтерфейсу здійснюється без перезавантаження сторінки, що досягається використанням асинхронних запитів та повторного рендерингу DOM-структури.
	Додатково на стороні клієнта виконуються обчислювальні операції, зокрема розрахунок середнього балу на основі отриманих оцінок:
const avg = (grades.reduce((a, g) => a + g.value, 0) / grades.length).toFixed(2);
	Результат одразу відображається в інтерфейсі користувача.

	4.4 Архітектурна взаємодія компонентів системи
	Загальна взаємодія компонентів вебзастосунку базується на клієнт-серверній архітектурі. Клієнтська частина відповідає за відображення інтерфейсу та обробку дій користувача, тоді як серверна частина забезпечує зберігання та обробку даних.
	У системі можна виділити такі логічні компоненти: інтерфейс користувача (HTML), логіка взаємодії (JavaScript), механізм збереження локальних даних (localStorage) та серверний REST API.
	Взаємодія між цими компонентами відбувається за наступним принципом: користувач взаємодіє з інтерфейсом, JavaScript обробляє події та формує запити до API, сервер повертає дані у форматі JSON, після чого вони відображаються у DOM-структурі сторінки.
	Такий підхід забезпечує модульність системи, спрощує її масштабування та дозволяє незалежно змінювати клієнтську та серверну частини без порушення загальної логіки роботи застосунку.

Розділ 6. Висновки
	У межах виконання лабораторної роботи було розроблено клієнтську частину вебзастосунку, призначеного для організації взаємодії користувачів різних ролей у навчальній системі. Реалізований вебінтерфейс забезпечує роботу в режимі клієнт–сервер та інтегрується з попередньо створеною серверною частиною через REST API.
	У ході виконання роботи було розроблено набір окремих інтерфейсів для різних категорій користувачів, а саме: студента, викладача, батьків та адміністратора. Кожен інтерфейс має власну логіку відображення даних та функціональні можливості, що відповідають ролі користувача в системі. Це дозволило реалізувати розмежування доступу та забезпечити логічну структуризацію функціоналу.                       
	Реалізовано механізм авторизації користувачів із подальшим розподілом за ролями та маршрутизацією до відповідних сторінок інтерфейсу. Стан користувача після входу зберігається на стороні клієнта, що дозволяє підтримувати сесію без повторної авторизації під час оновлення сторінки.
Важливою частиною роботи є взаємодія з серверною частиною через HTTP-запити. Дані отримуються у форматі JSON та динамічно обробляються на стороні клієнта. Це забезпечує актуальність відображуваної інформації та дозволяє оновлювати інтерфейс без перезавантаження сторінок.
	Окремо реалізовано використання локального сховища браузера для збереження допоміжних даних, таких як ідентифікатори користувачів, фотографії профілю та персональні налаштування. Це підвищує зручність користування системою та частково зменшує залежність від серверної частини.
Результатом виконання лабораторної роботи є функціональний вебзастосунок із розподіленими ролями користувачів, динамічним інтерфейсом та реалізованою взаємодією з серверним API.

        localStorage.setItem("role", "admin");

        window.location.href = "admin.html";
        return;
    }

    switch (role) {
        case "student":
            url = `https://localhost:7033/api/students/byEmail?email=${encodeURIComponent(email)}`;
            break;
        case "teacher":
            url = `https://localhost:7033/api/teachers/byEmail?email=${encodeURIComponent(email)}`;
            break;
        case "parent":
            url = `https://localhost:7033/api/parents/byEmail?email=${encodeURIComponent(email)}`;
            break;
    }

    const response = await fetch(url);
    const data = await response.json();

    localStorage.setItem("role", role);
});

Б.2 – Завантаження та відображення оцінок студента

function loadGradesFromProfile(grades) {
    const table = document.getElementById("gradesTable");
    table.innerHTML = "";

    if (!grades || grades.length === 0) {
        table.innerHTML = `<tr><td colspan="4">Оцінок поки що немає</td></tr>`;
        return;
    }

    grades.forEach(g => {
        const row = `
            <tr>
                <td>${g.disciplineName || "-"}</td>
                <td>${g.value}</td>
                <td>${g.teacherName || "-"}</td>
                <td>${g.date ? new Date(g.date).toLocaleDateString() : "-"}</td>
            </tr>
        `;
        table.innerHTML += row;
    });
}
