	2 ЗАВДАННЯ
	Обрати шаблон (патерн) проєктування ПЗ із книги GoF (крім Singleton). Підготувати приклади програмного коду, що демонструють використання цього патерна, та надати детальні пояснення щодо його застосування, зокрема:
	- основне призначення патерна та задачі, які він вирішує;

	- передумови доцільності застосування;

	3 ОПИС ВИКОНАНОЇ РОБОТИ
	Для виконання завдання обрано патерн Observer, який належить до поведінкових патернів проектування. Основна мета роботи — дослідити спосіб організації взаємодії між об’єктами таким чином, щоб зміни стану одного об’єкта автоматично впливали на інші об’єкти без створення жорсткого зв’язку між ними. Такий підхід дозволяє зробити програмне забезпечення більш гнучким, масштабованим та підтримуваним, що особливо важливо для сучасних складних програмних систем.
	Під час роботи було проведено аналіз теоретичних аспектів патерна Observer, вивчено практичні приклади його реалізації, порівняно з іншими патернами, а також розроблено демонстраційну програму на мові C#. Завдяки такому комплексному підходу вдалося не лише зрозуміти основи патерна, а й сформувати навички практичного його застосування у реальних системах.

	3.1. Основні компоненти патерна Observer
	Патерн Observer базується на двох ключових елементах — Subject і Observer.
	Subject (суб’єкт) — це об’єкт, стан якого потрібно відстежувати. Він зберігає список усіх спостерігачів та відповідає за їх повідомлення про зміни стану через метод notifyObservers(). Важливо, що суб’єкт не знає деталей реалізації спостерігачів, що забезпечує низьку зв’язаність системи. Така структура дозволяє легко додавати нові компоненти, не змінюючи логіку основного об’єкта, що особливо актуально у великих програмних системах.
	Observer (спостерігач) — об’єкт, який підписується на зміни стану суб’єкта. Кожен Observer реалізує метод update(), у якому описано, як реагувати на повідомлення. Спостерігач може виконувати різні дії: оновлювати дані на екрані, вести логування, запускати додаткові процеси або змінювати стан інших компонентів системи.
	Методи взаємодії — notifyObservers() у суб’єкта викликає метод update() кожного спостерігача. Це забезпечує автоматичне поширення змін без прямого зв’язку між об’єктами. Завдяки цьому можна динамічно додавати нових спостерігачів, не змінюючи код суб’єкта, що значно підвищує масштабованість та підтримуваність системи.
	Observer дозволяє ефективно реалізовувати патерн підписки-повідомлення, де кожен модуль системи отримує необхідну інформацію про зміни і може реагувати відповідно до своїх функцій. Це особливо корисно для складних додатків, де багато компонентів повинні синхронно обробляти події.	
	3.2. Приклад реалізації на C#
	Для демонстрації патерна Observer було розроблено просту програму на C#, яка ілюструє принцип роботи патерна.
	Інтерфейс IObserver визначає метод Update(string msg), який отримує повідомлення від суб’єкта.
	Клас ConcreteObserver реалізує інтерфейс і виводить повідомлення в консоль, демонструючи реакцію на зміну стану суб’єкта.
	Клас Subject зберігає список спостерігачів і надає методи Add() для додавання нового Observer та Notify() для сповіщення всіх спостерігачів.
	Цей приклад показує, що спостерігачі можуть діяти автономно, не знаючи деталей реалізації суб’єкта. Основна ідея полягає у гнучкій архітектурі, де додавання нових спостерігачів не потребує змін у коді суб’єкта, а всі повідомлення обробляються централізовано через стандартний метод.
	Реалізація на C# демонструє чітке розділення обов’язків: суб’єкт відповідає лише за сповіщення, а Observer — за обробку події. Це дозволяє легко тестувати окремі компоненти і забезпечує зрозумілу логіку програми, що особливо важливо для командної розробки.

	3.3. Призначення та практичне використання
	Патерн Observer застосовується для автоматичного сповіщення об’єктів про зміни стану іншого об’єкта, що дозволяє зменшити залежності між класами та підвищити гнучкість системи.
	Основне призначення — автоматичне повідомлення спостерігачів, без необхідності вручну перевіряти стан суб’єкта.
	Observer підтримує динамічне додавання нових спостерігачів, що дозволяє розширювати функціонал системи без зміни існуючого коду.
	Завдяки патерну можна підтримувати низьку зв’язаність між компонентами: спостерігачі не знають деталей реалізації суб’єкта, а суб’єкт не знає, які саме дії виконують Observer.
	Наприклад, у графічному інтерфейсі користувача натискання кнопки (суб’єкт) може одночасно оновлювати кілька модулів (спостерігачів). Кожен модуль реагує на подію по-своєму, але при цьому код кнопки не змінюється. Така архітектура дозволяє масштабувати систему, додаючи нові спостерігачі, не порушуючи роботу існуючих компонентів.
	Observer також широко використовується у системах підписки та сповіщень, наприклад, соціальні мережі або платформи новин автоматично повідомляють користувачів про оновлення контенту. Крім того, патерн застосовується для логування та моніторингу: різні модулі отримують повідомлення про помилки або зміну стану програми і реагують відповідно до своїх функцій.

	3.4. Порівняння з іншими патернами
	Observer відрізняється від інших поведінкових патернів:
	- Mediator централізовано координує взаємодію між об’єктами, тоді як Observer дозволяє об’єктам реагувати самостійно і незалежно один від одного.
	- Strategy змінює поведінку об’єкта через вибір алгоритму, але не повідомляє інші об’єкти про зміни стану.
	- Observer ефективний для розподілених сповіщень, де багато модулів повинні одночасно отримувати інформацію про зміни.
	Таким чином, патерн Observer є ключовим інструментом для створення гнучких і масштабованих систем, де додавання нових функцій не порушує роботу вже існуючих компонентів.
	3.5. Переваги використання
	Observer забезпечує кілька важливих переваг для розробки програмних систем:
    1) Масштабованість системи — легко додавати нові спостерігачі без зміни логіки суб’єкта.
    2) Повторне використання коду — окремі Observer можна застосовувати у різних частинах системи або навіть у інших проєктах.
    3) Підтримка принципу відкритості/закритості — класи відкриті для розширення, але закриті для модифікації.
    4) Підвищена підтримуваність — додавання нових функцій не потребує переробки існуючих модулів, що значно економить час розробки.
	Observer дозволяє організувати систему так, щоб зміни одного елемента автоматично впливали на інші, при цьому зберігаючи прозорість логіки та чіткий розподіл відповідальності між компонентами.

	3.6. Приклад реалізації патерна Observer
	Для демонстрації роботи патерна Observer було створено невеликий проект на мові C#. Проєкт складається з чотирьох компонентів: Subject (суб’єкт), IObserver (інтерфейс спостерігача), ConcreteObserver (конкретний спостерігач) та головної програми Program.
	Основна логіка роботи:
	- Subject зберігає список підписаних спостерігачів і відповідає за їх повідомлення при зміні стану.
	- Observer визначає метод Update(), який кожен спостерігач реалізує відповідно до власних потреб.
	- ConcreteObserver реалізує інтерфейс Observer і визначає конкретну реакцію на повідомлення (наприклад, вивід у консоль).
	- У головній програмі створюється об’єкт Subject, додаються ConcreteObserver, і викликається метод Notify(), що сповіщає всіх спостерігачів про зміну стану.
	Цей приклад демонструє основні переваги патерна Observer: низьку зв’язаність, масштабованість, гнучкість та можливість динамічного додавання спостерігачів без зміни логіки суб’єкта.
	Повний приклад програмного коду наведено у Додатку В – Програмний код.

	4 ВИСНОВКИ
	У результаті виконаної роботи було детально опрацьовано патерн проєктування Observer, його структуру та принципи реалізації. Було досліджено ключові компоненти патерна — Subject, Observer та взаємодію між ними при зміні стану об’єкта. Це дозволило зрозуміти, як організувати автоматичне сповіщення об’єктів та забезпечити низьку зв’язаність між компонентами системи, що є основною метою патерна Observer.
	Особлива увага приділялася принципам чистого коду та структурованої організації програмних модулів. Було проаналізовано важливість логічного групування методів, правильного підключення залежностей та організації даних у класах і структурах. Використання зрозумілих імен для об’єктів та методів дозволяє одразу визначати призначення коду, що спрощує підтримку, читання та командну роботу над проєктом.
	В рамках роботи також розглянуто методи оптимізації продуктивності та обробки помилок. Було показано, як ефективно управляти колекціями об’єктів спостерігачів, мінімізувати дублювання коду шляхом виділення повторюваних операцій у окремі методи та організовувати контроль некоректних станів для забезпечення стабільності системи. Це дозволяє створювати надійні та передбачувані програмні рішення.
	Крім того, виконано аналіз дотримання принципів об’єктно-орієнтованого програмування та модульності. Створення окремих класів для суб’єктів і спостерігачів, визначення спільної поведінки через інтерфейси або абстрактні класи, а також дотримання принципу «один метод — одна відповідальність» сприяє повторному використанню коду, зменшенню взаємозалежності компонентів і полегшує масштабування системи.
	Особливу увагу приділено тестуванню та документуванню. Перевірка роботи патерна на різних сценаріях показала, як спостерігачі реагують на зміни стану суб’єкта, а коментарі та описи логіки методів забезпечують зрозумілість і прозорість роботи програмних блоків для інших розробників. Це значно полегшує подальшу підтримку і розвиток системи.
	Загалом, виконана робота дозволила ознайомитися з теоретичними та практичними аспектами патерна Observer, навчитися організовувати взаємодію об’єктів у програмних системах, створювати гнучкі, модульні та підтримувані рішення. Отримані знання формують професійні навички програміста, допомагають створювати структуровані та надійні системи, готові до масштабування та розширення у будь-яких проєктах.

В.1 Приклад коду патерна Observer (C#) 

1  using System;
2  using System.Collections.Generic;
3
4  interface IObserver { 
5      void Update(string msg); // метод отримання повідомлень
6  }
7
8  class ConcreteObserver : IObserver {
9      string name;
10     public ConcreteObserver(string n){ name = n; } // задаємо ім’я
11     public void Update(string msg){ Console.WriteLine(name + ": " + msg); } // вивід повідомлення
12 }
13
14 class Subject {
15     List<IObserver> observers = new List<IObserver>(); // список спостерігачів
16
17     public void Add(IObserver o){ observers.Add(o); } // додати спостерігача
18     public void Notify(string msg){ observers.ForEach(o => o.Update(msg)); } // повідомити всіх
19 }
20
21 class Program {
22     static void Main() {
23         var s = new Subject(); // створюємо суб’єкт
24         s.Add(new ConcreteObserver("A")); // додаємо спостерігача A
25         s.Add(new ConcreteObserver("B")); // додаємо спостерігача B
26         s.Notify("Стан змінено"); // повідомляємо всіх спостерігачів
27     }
28 }
