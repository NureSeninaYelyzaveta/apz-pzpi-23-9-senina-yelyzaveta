Розділ 1. Аналіз предметної області та постановка задачи 
	1.1 Призначення мобільного застосунку
	Сучасні інформаційні системи все частіше використовують мобільні програмні застосунки як окрему частину загальної програмної системи. Використання мобільних платформ дозволяє забезпечити швидкий доступ користувачів до інформації незалежно від місця перебування та часу використання системи. Особливо актуальним це є для освітніх платформ, у яких необхідно забезпечити оперативний доступ до навчальної інформації, оцінок, результатів успішності та взаємодії між учасниками освітнього процесу.
	У межах лабораторної роботи було розроблено мобільний застосунок для операційної системи Android, який є клієнтською частиною інформаційної системи Art Progress. Основним призначенням застосунку є забезпечення взаємодії користувачів із серверною частиною системи через мобільний інтерфейс.
	Мобільний застосунок реалізований мовою програмування Kotlin із використанням сучасних засобів розробки Android-застосунків. Для побудови графічного інтерфейсу використано Jetpack Compose, а для організації взаємодії із серверною частиною системи — бібліотеку Retrofit та REST API.
	Система підтримує декілька категорій користувачів, кожна з яких має власний функціонал та набір доступних можливостей. У межах мобільного застосунку реалізовано роботу для таких ролей:
	- учень;
	- батьки;
	- викладач.
	Учень має можливість переглядати власні оцінки, інформацію про дисципліни та викладачів. Батьки можуть переглядати інформацію про успішність дитини та отримувати доступ до її оцінок через зв’язок між сутностями системи. Викладач має доступ до інформації, пов’язаної з навчальним процесом та оцінюванням студентів.
	Мобільний застосунок функціонує як окремий клієнтський модуль системи та взаємодіє із серверною частиною за допомогою HTTP-запитів. Обмін даними між клієнтом та сервером здійснюється у форматі JSON.















Рис. 1 – Загальна схема взаємодії мобільного застосунку із серверною частиною системи

	1.2 Основні функції системи
	Розроблений мобільний застосунок забезпечує доступ користувачів до функціональних можливостей інформаційної системи Art Progress через мобільний пристрій.
	Основною функцією застосунку є авторизація користувачів та подальше надання доступу до відповідного функціоналу залежно від ролі користувача. Після входу до системи користувач отримує доступ лише до тих даних та можливостей, які відповідають його типу облікового запису.
	Для учнів реалізовано такі функціональні можливості:
	- авторизація за електронною поштою;
	- перегляд особистої інформації;
	- перегляд списку оцінок;
	- перегляд дисциплін;
	- перегляд інформації про викладачів;
	- отримання даних із серверної частини системи.
	Для батьків реалізовано:
	- авторизацію в системі;
	- отримання інформації про пов’язаних учнів;
	- перегляд оцінок дитини;
	- перегляд інформації про дисципліни;
	- перегляд інформації про викладачів.
	Для викладачів реалізовано:
	- авторизацію;
	- перегляд навчальної інформації;
	- доступ до даних про студентів та оцінювання.
	Особливістю реалізованої системи є наявність взаємозв’язку між сутностями батьків та учнів через проміжну таблицю ParentStudent. Це дозволяє автоматично визначати, які саме учні пов’язані з конкретним обліковим записом батьків, без необхідності додаткового введення інформації користувачем.
	Отримання даних виконується шляхом звернення мобільного застосунку до REST API серверної частини системи. Для обробки мережевих запитів використовується асинхронний механізм взаємодії, що забезпечує коректну роботу користувацького інтерфейсу без блокування основного потоку виконання програми.
	З метою підвищення зручності використання системи та розподілу функціональних обов’язків між клієнтськими частинами, було прийнято рішення обмежити функціонал мобільного застосунку переважно режимом перегляду даних. Функції додавання та редагування навчальної інформації, зокрема виставлення оцінок та управління портфоліо, реалізуються у веб-версії системи, яка є більш зручною для роботи викладачів та адміністраторів. Мобільний застосунок орієнтований на швидкий доступ до навчальної інформації та моніторинг успішності користувачів. 












Рис. 2 – UML-діаграма прецедентів мобільного застосунку

	1.3 Ролі користувачів у системі
	У мобільному застосунку реалізовано багаторівневу систему доступу, яка базується на ролях користувачів. Кожна роль має окремий набір функціональних можливостей та доступ до відповідних даних системи.
	Учень
	Учень є основним користувачем системи. Після проходження авторизації користувач отримує доступ до власної навчальної інформації. До функціоналу учня входить перегляд оцінок, дисциплін та інформації про викладачів.
	Дані учня завантажуються із серверної частини системи через API-запити. Інформація відображається у мобільному інтерфейсі у структурованому вигляді.
	Батьки
	Роль батьків призначена для контролю успішності дитини. Після авторизації система автоматично визначає учнів, пов’язаних із відповідним обліковим записом батьків.
	Для реалізації такого механізму використовується проміжна таблиця ParentStudent, яка забезпечує зв’язок між сутностями Parent та Student. Після отримання списку зв’язків застосунок виконує запити до серверної частини системи для отримання інформації про кожного учня та його оцінки.
	Завдяки такому підходу батькам не потрібно додатково вводити електронну пошту або інші дані дитини для перегляду інформації.
	Викладач
	Викладач має доступ до інформації, пов’язаної з навчальним процесом. У межах реалізованого функціоналу викладач може отримувати дані із серверної частини системи та переглядати інформацію, необхідну для роботи з оцінюванням студентів.
	Для кожної ролі реалізовано окремі екрани мобільного застосунку та окрему логіку роботи інтерфейсу. Це дозволяє забезпечити зручність використання системи та розмежування доступу до даних.

	1.4 Постановка задачі
	Метою лабораторної роботи є розробка мобільного застосунку для платформи Android, який взаємодіє із серверною частиною інформаційної системи Art Progress.
	Під час виконання роботи необхідно було реалізувати:
	- систему авторизації користувачів;
	- підтримку декількох ролей користувачів;
	- взаємодію із REST API;
	- отримання та обробку даних у форматі JSON;
	- відображення інформації у графічному інтерфейсі;
	- навігацію між екранами застосунку;
	- архітектурну організацію проєкту;
	- взаємодію між мобільним клієнтом та серверною частиною системи.
	Окрему увагу було приділено реалізації архітектури застосунку та розподілу логіки між компонентами системи. Для цього було використано архітектурний підхід MVVM, який забезпечує розділення інтерфейсу користувача, бізнес-логіки та роботи з даними.
	У результаті виконання лабораторної роботи було створено мобільний застосунок, який забезпечує взаємодію користувачів із серверною системою та реалізує основні функції освітньої інформаційної платформи.

Розділ 2. Проектування мобільного застосунку
	2.1 Вибір технологій розробки
	Під час проєктування мобільного застосунку для інформаційної системи Art Progress було обрано сучасний стек технологій, який забезпечує стабільну роботу клієнтської частини, зручну взаємодію із сервером та можливість подальшого розширення функціоналу.
	Основною платформою розробки обрано Android, оскільки вона є найбільш поширеною серед мобільних пристроїв та надає широкий набір інструментів для створення сучасних застосунків.
	Як основну мову програмування використано Kotlin, який є офіційно рекомендованим для Android-розробки. Його використання дозволяє зменшити обсяг коду, підвищити його читабельність та забезпечити безпечну роботу з даними.
	Для створення графічного інтерфейсу застосовано Jetpack Compose, що дозволило реалізувати декларативний підхід до побудови UI та спростити керування станами екранів.
	Взаємодія із серверною частиною системи реалізована за допомогою бібліотеки Retrofit, яка забезпечує роботу з REST API та автоматичну десеріалізацію JSON-відповідей у Kotlin-об’єкти.
	Для асинхронної обробки мережевих запитів використано Kotlin Coroutines, що дозволяє виконувати запити без блокування головного потоку інтерфейсу.
	Архітектура застосунку побудована на основі патерну MVVM (Model–View–ViewModel), що забезпечує розділення логіки, представлення та роботи з даними.
	Середовище розробки — Android Studio, яке використовується для реалізації, тестування та налагодження застосунку.

	2.2 Архітектура мобільного застосунку
	Архітектура мобільного застосунку побудована на основі патерну MVVM, який дозволяє розділити систему на логічні рівні та спростити її підтримку.
	У межах реалізації виділено такі основні компоненти:
	- рівень представлення (UI);
	- рівень бізнес-логіки (ViewModel);
	- рівень доступу до даних (Network layer);
	- рівень моделей даних (DTO).
	Рівень представлення відповідає за відображення інтерфейсу користувача та реалізований за допомогою Jetpack Compose.
	ViewModel виконує роль посередника між UI та даними, зберігає стан екранів і обробляє бізнес-логіку.
	Мережевий рівень реалізований через Retrofit API Service, який відповідає за HTTP-запити до серверної частини системи.
	Для передачі даних використано DTO-моделі, які відповідають структурі JSON-відповідей сервера.


















Рис. 3 – Архітектура мобільного застосунку (MVVM)
	2.3 Організація структури проєкту
	Структура мобільного застосунку побудована за модульним (пакетним) принципом, що дозволяє логічно розділити відповідальність між компонентами системи.
	Проєкт поділено на такі основні пакети:
	1. ui                                         
	Містить екрани та компоненти інтерфейсу користувача, реалізовані за допомогою Jetpack Compose. Тут знаходяться екрани авторизації, а також інтерфейси для кожної ролі користувача.
	2. viewmodel                                    
	Містить класи, які керують станом інтерфейсу та обробляють бізнес-логіку взаємодії між UI та сервером.
	3. network                                      
	Відповідає за реалізацію мережевої взаємодії. Тут знаходяться Retrofit-клієнт, API-інтерфейси та конфігурація запитів.
	4. dto                                            
	Містить моделі даних, які використовуються для обміну інформацією із сервером (StudentDto, ParentDto, TeacherDto, GradeDto тощо).
	Навігація між екранами реалізована через Navigation Compose та винесена в окремий компонент AppNav.























Рис. 4 – Структура пакетів мобільного застосунку

	2.4 Модель даних системи
	Серверна частина системи Art Progress використовує реляційну модель даних, з якою взаємодіє мобільний застосунок.
	Основними сутностями є:
	- Student;
	- Parent;
	- Teacher;
	- Grade;
	- Discipline;
	- ParentStudent.
	Сутність Student містить інформацію про учнів (ім’я, група, контактні дані).
	Сутність Parent зберігає дані батьків.
	Сутність Teacher містить інформацію про викладачів.
	Сутність Grade відображає оцінки студентів із прив’язкою до дисципліни та викладача.
	Для реалізації зв’язку між батьками та учнями використовується проміжна таблиця ParentStudent, яка реалізує зв’язок типу “багато до багатьох”.
	У мобільному застосунку дані передаються через DTO-об’єкти, що забезпечує незалежність клієнта від внутрішньої структури серверної бази даних.

















Рис. 5 – ER-діаграма моделі даних системи

	2.5 Організація взаємодії із серверною частиною системи
	Взаємодія мобільного застосунку із серверною частиною системи Art Progress реалізована через REST API.
	Обмін даними виконується за допомогою HTTP-запитів, а формат передачі даних — JSON.
	Основні типи запитів:
    1. отримання даних користувача;
    2. авторизація;
    3. отримання списку студентів;
    4. отримання оцінок;
    5. отримання зв’язків ParentStudent;
    6. отримання даних викладачів.
	Для реалізації HTTP-запитів використовується бібліотека Retrofit, яка автоматично виконує серіалізацію та десеріалізацію даних.
	Асинхронна обробка запитів реалізована за допомогою Kotlin Coroutines, що дозволяє уникнути блокування інтерфейсу користувача.
	У разі помилок мережевого з’єднання передбачено базову обробку виключень, що підвищує стабільність роботи застосунку.































Рис. 6 – Схема взаємодії мобільного застосунку з REST API


Розділ 3. Реалізація мобільного застосунку
	3.1 Реалізація авторизації користувачів
	У межах мобільного застосунку реалізовано механізм авторизації користувачів, який є базовою точкою входу до системи. Авторизація здійснюється на основі електронної пошти користувача та вибору ролі: учень, батьки або викладач.
	Такий підхід був обраний як спрощений варіант автентифікації для навчального проєкту. Він дозволяє зосередитися на реалізації клієнт-серверної взаємодії та логіки розмежування доступу між різними типами користувачів без впровадження складних механізмів безпеки (паролі, токени доступу, JWT тощо).
	Після введення електронної пошти та вибору ролі виконується запит до відповідного API-методу серверної частини системи. У разі успішної відповіді користувач перенаправляється на відповідний екран залежно від ролі, а також отримує унікальний ідентифікатор, який використовується для подальшої роботи із даними.
	У перспективі розвитку системи передбачається розширення механізму авторизації шляхом додавання пароля, реєстрації користувачів, підтвердження електронної пошти та впровадження більш безпечної системи автентифікації.

	Для забезпечення коректної роботи механізму авторизації у мобільному застосунку реалізовано послідовність взаємодії між користувачем, інтерфейсом застосунку та серверною частиною системи. Під час входу користувач вводить електронну пошту та обирає відповідну роль, після чого застосунок формує HTTP-запит до REST API. Сервер виконує перевірку наявності користувача та повертає відповідь із даними облікового запису. На основі отриманої ролі система автоматично відкриває відповідний екран мобільного застосунку. Послідовність виконання цих дій представлена на UML-діаграмі діяльності процесу авторизації користувача. 






















Рис. 9 – UML-діаграма діяльності процесу авторизації користувача 
	3.2 Реалізація взаємодії з роллю “Батьки”
	Роль батьків у мобільному застосунку призначена для перегляду навчальної інформації про пов’язаних учнів. Після авторизації система визначає ідентифікатор батьківського акаунта та використовує його для отримання відповідних даних із серверної частини системи.
	Взаємодія між батьками та учнями реалізована через проміжну сутність ParentStudent, яка забезпечує зв’язок типу «багато до багатьох». Це дозволяє одному обліковому запису батьків бути пов’язаним з одним або декількома учнями.
	Алгоритм отримання даних включає наступні етапи:
	- отримання даних про батьків за електронною поштою;
	- завантаження списку зв’язків ParentStudent;
	- фільтрація записів за ідентифікатором батьків;
	- отримання даних учнів, пов’язаних із даним користувачем;
	- відображення інформації про учнів та їхні оцінки у користувацькому інтерфейсі.
	Таким чином, реалізований підхід дозволяє автоматично отримувати дані про дітей без необхідності додаткового введення інформації користувачем, що підвищує зручність використання системи.

	3.3 Реалізація інтерфейсу мобільного застосунку для різних ролей
	Мобільний застосунок побудовано за принципом розділення інтерфейсів відповідно до ролей користувачів. Кожна роль має власний екран, який відображає лише ту інформацію, що відповідає рівню доступу користувача.
	Роль учня
	Учень після авторизації отримує доступ до особистої навчальної інформації. Основними функціями є перегляд оцінок, дисциплін та інформації про викладачів. Дані отримуються із серверної частини системи та відображаються у вигляді списків у зручному для користувача форматі.
	Роль батьків
	Батьки отримують доступ до інформації про пов’язаних учнів та їхні навчальні результати. Інтерфейс дозволяє переглядати список дітей, їхні оцінки та інформацію про навчальні дисципліни.
	Роль викладача
	Викладач має доступ до навчальної інформації, що стосується студентів та процесу оцінювання. Інтерфейс цієї ролі орієнтований на перегляд даних, пов’язаних із навчальним процесом.
	Таким чином, реалізована система інтерфейсів забезпечує логічне розділення функціоналу між користувачами та підвищує зручність роботи із застосунком.

	Для забезпечення логічної організації роботи мобільного застосунку було реалізовано послідовність переходів між основними станами системи. UML-діаграма станів демонструє основні етапи роботи застосунку та переходи між екранами під час взаємодії користувача із системою.















Рис. 13 – UML-діаграма станів мобільного застосунку 
	3.4 Реалізація клієнт-серверної взаємодії
	Мобільний застосунок взаємодіє із серверною частиною системи через REST API. Обмін даними здійснюється у форматі JSON за допомогою HTTP-запитів.
	Для реалізації мережевої взаємодії використано бібліотеку Retrofit, яка забезпечує зручне визначення API-інтерфейсів та автоматичну десеріалізацію даних у Kotlin-об’єкти (DTO).
	Усі мережеві запити виконуються асинхронно із використанням Kotlin Coroutines, що дозволяє уникнути блокування головного потоку застосунку та забезпечує стабільну роботу інтерфейсу.
	Основні типи запитів, які використовуються у застосунку:
	- отримання даних про користувача за електронною поштою;
	- отримання інформації про учнів, батьків та викладачів;
	- отримання зв’язків між сутностями системи;
	- отримання оцінок та навчальної інформації.
	Такий підхід забезпечує гнучку архітектуру клієнтської частини та дозволяє легко масштабувати систему у майбутньому.

Розділ 4. Висновки
	У межах виконання лабораторної роботи було розроблено мобільний застосунок для операційної системи Android, який є клієнтською частиною інформаційної системи Art Progress та забезпечує взаємодію користувачів із серверною частиною через REST API.
	Під час розробки застосунку було використано сучасні технології мобільної розробки, зокрема мову програмування Kotlin, інструментарій Jetpack Compose для побудови користувацького інтерфейсу, бібліотеку Retrofit для організації мережевої взаємодії та формат обміну даними JSON. Для реалізації асинхронної обробки запитів застосовано Kotlin Coroutines, що забезпечило стабільну роботу інтерфейсу без блокування основного потоку виконання.
	Архітектура мобільного застосунку побудована на основі патерну MVVM, що дозволило розділити рівні представлення, бізнес-логіки та роботи з даними. Це забезпечило структурованість коду, його масштабованість та зручність подальшої підтримки.
	Реалізований застосунок підтримує декілька ролей користувачів, зокрема учня, батьків та викладача. Для кожної ролі визначено окремий інтерфейс та набір функціональних можливостей, що забезпечує розмежування доступу до даних відповідно до ролі користувача в системі.
	Основним функціоналом мобільного застосунку є авторизація користувачів та перегляд навчальної інформації, зокрема оцінок та даних про учнів. Система взаємодіє із серверною частиною через REST API та отримує дані у форматі JSON, що забезпечує уніфікований підхід до обміну інформацією.
	Реалізований підхід до побудови мобільного застосунку дозволяє забезпечити зручний доступ до навчальної інформації та може бути розширений у майбутньому шляхом додавання функцій редагування даних, розширеної системи автентифікації та підтримки додаткових модулів освітньої платформи.

fun loadStudent(id: Int) {
    viewModelScope.launch {
        state.value = state.value.copy(loading = true)

        val response = RetrofitClient.apiService.getStudent(id)

        state.value = state.value.copy(
            student = response,
            loading = false
        )
    }
}
